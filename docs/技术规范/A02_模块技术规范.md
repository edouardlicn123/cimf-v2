## 一、概述

### 1.1 核心概念

| 概念 | 说明 |
|------|------|
| **模块 (Module)** | 独立的业务功能单元，每个模块对应一个 Django 子应用 |
| **节点类型 (NodeType)** | 模块的业务分类，对应事务处理页面的入口 |
| **节点 (Node)** | 节点的公共字段，记录创建人、更新时间等元信息 |
| **字段模型 ({Module}Fields)** | 模块的业务数据模型，存储具体业务字段 |

### 1.2 模块类型

模块通过 `type` 字段分为三种类型：

| 类型 | 说明 | 创建类型 | 显示位置 |
|------|------|----------|----------|
| `node` | 事务节点模块，用于业务数据管理 | `NodeType` | 事务处理页面（首页、左侧导航） |
| `system` | 系统模块，用于系统功能扩展 | 无 | 仪表盘、功能卡片等 |
| `tool` | 协作工具模块，用于工具类功能 | `ToolType` | 协作工具页面（首页、左侧导航） |

**设计理念**：
- `node` 类型模块会创建 `NodeType`，出现在事务处理相关页面
- `system` 类型模块不参与类型管理，不出现在事务处理或协作工具页面
- `tool` 类型模块会创建 `ToolType`，出现在协作工具相关页面，包括工具首页卡片式展示

```
                                    │
```

## 三、module.py 规范

### 3.1 必选字段

```python
# -*- coding: utf-8 -*-
"""
模块信息文件
定义模块的基本信息，供系统扫描和注册使用。
"""

MODULE_INFO = {
    'id': 'customer',                    # 模块唯一标识
    'name': '客户信息（海外）',            # 显示名称
    'type': 'node',                      # 模块类型 (node=事务节点, system=系统模块)
    'version': '1.0.0',                  # 版本号
}
```

### 3.2 可选字段

| 字段 | 类型 | 说明 | 详细规范 |
|------|------|------|----------|
| `author` | string | 作者 | - |
| `description` | string | 描述 | - |
| `icon` | string | Bootstrap Icons 类名，如 `bi-people` | - |
| `require` | list[string] | 依赖模块列表 | [3.7 节](#37-require-字段规范) |
| `dashboard_cards` | list | 仪表盘卡片配置 | [3.5 节](#35-dashboard_cards-字段规范) |
| `taxonomies` | list | 模块词汇表定义，安装时自动创建 | [3.3 节](#33-taxonomies-字段规范) |
| `permissions` | list | 模块自定义权限 | [3.4 节](#34-permissions-字段规范) |
| `export_fields` | list | 导出字段配置 | [3.6 节](#36-export_fields-字段规范) |
| `dashboard_stats` | bool | 是否支持仪表盘统计 | - |
| `views` | dict | 视图函数映射 | - |

### 3.3 taxonomies 字段规范

模块可在 `module.py` 中定义词汇表，系统在安装模块时会自动创建这些词汇表及词汇项。

**数据结构**：

```python
'taxonomies': [
    {
        'slug': 'supplier_type',           # 必填，词汇表唯一标识（全局唯一）
        'name': '供应商类型',                # 必填，词汇表显示名称
        'items': ['原材料', '设备', '服务']  # 必填，词汇项名称列表
    },
]
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `slug` | string | 是 | 词汇表唯一标识，全局唯一 |
| `name` | string | 是 | 词汇表显示名称 |
| `items` | list[string] | 是 | 词汇项名称列表 |

**安装行为**：

| 场景 | 行为 |
|------|------|
| 词汇表不存在 | 创建词汇表 + 创建所有词汇项 |
| 词汇表已存在 | 跳过创建，仅补充缺失的词汇项 |
| 词汇项已存在 | 跳过，不重复创建 |

**示例**：

```python
MODULE_INFO = {
    'id': 'supplier',
    'name': '供应商',
    'type': 'node',
    'version': '1.0.0',
    'taxonomies': [
        {'slug': 'supplier_type', 'name': '供应商类型', 'items': ['原材料', '设备', '服务']},
        {'slug': 'supplier_level', 'name': '供应商等级', 'items': ['A级', 'B级', 'C级']},
    ],
}
```

### 3.4 permissions 字段规范

模块可在 `module.py` 中定义自定义权限，系统会自动将这些权限加入权限管理页面。

**数据结构**：

```python
'permissions': [
    {'key': 'view_others', 'name': '查看别人的内容'},
    {'key': 'edit_others', 'name': '修改别人的内容'},
    {'key': 'delete_others', 'name': '删除别人的内容'},
]
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `key` | string | 是 | 权限标识，最终权限为 `node.{module_id}.{key}` |
| `name` | string | 是 | 权限显示名称 |

**权限生成规则**：

| 权限类型 | 权限标识 | 说明 |
|----------|----------|------|
| 基础权限 | `node.{module_id}.create` | 创建节点（自动添加） |
| 基础权限 | `node.{module_id}.read` | 查看节点（自动添加） |
| 基础权限 | `node.{module_id}.update` | 修改节点（自动添加） |
| 基础权限 | `node.{module_id}.delete` | 删除节点（自动添加） |
| 自定义权限 | `node.{module_id}.{key}` | 从配置读取 |

**示例**：

```python
MODULE_INFO = {
    'id': 'supplier',
    'name': '供应商',
    'type': 'node',
    'version': '1.0.0',
    'icon': 'bi-box-seam',
    'permissions': [
        {'key': 'view_others', 'name': '查看别人的内容'},
        {'key': 'edit_others', 'name': '修改别人的内容'},
        {'key': 'delete_others', 'name': '删除别人的内容'},
    ],
}
```

**安装行为**：

| 场景 | 行为 |
|------|------|
| 模块已启用 | 权限自动出现在权限管理页面 |
| 模块未启用 | 权限不显示 |

### 3.5 dashboard_cards 字段规范

模块可在 `module.py` 中定义功能卡片配置，系统会在仪表盘显示对应的快捷入口。

**数据结构**：

```python
'dashboard_cards': [
    {
        'id': 'supplier_card',                    # 必填，卡片唯一标识
        'name': '供应商',                          # 必填，卡片显示名称
        'template': 'supplier/dashboard_card.html' # 必填，模板路径
    }
]
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 卡片唯一标识 |
| `name` | string | 是 | 卡片显示名称 |
| `template` | string | 是 | 模板路径（相对于模块 templates 目录） |
| `color_start` | string | 否 | 渐变起始颜色（HEX），如 #0d6efd |
| `color_end` | string | 否 | 渐变结束颜色（HEX），如 #0a58ca |

**模板中使用颜色变量**：

在模板中可通过以下变量使用卡片颜色：
```html
style="background: linear-gradient(135deg, {{ module_card_color_start }} 0%, {{ module_card_color_end }} 100%);"
```

**统计方法**：如需在卡片上显示数据，需在 Service 类中实现：

```python
@staticmethod
def get_count() -> int:
    """获取数据总数"""
    return Model.objects.count()

@staticmethod
def get_recent_count(days: int = 7) -> int:
    """获取最近N天新增数量"""
    from django.utils import timezone
    from datetime import timedelta
    start_date = timezone.now() - timedelta(days=days)
    return Model.objects.filter(created_at__gte=start_date).count()
```

**字段说明**：

| `name` | string | 是 | 数据库字段名 |
| `label` | string | 是 | 显示名称 |
| `type` | string | 是 | 字段类型 |
| `required` | bool | 否 | 是否必填 |

**字段类型**：

| 类型 | 说明 | 处理方式 |
|------|------|----------|

| `email` | 邮箱 | 直接导出 |
| `telephone` | 电话 | 直接导出 |
| `link` | 链接 | 直接导出 |

### 3.7 require 字段规范

模块可在 `module.py` 中声明依赖的其他模块。系统会在安装/启用时验证依赖是否满足。

**数据结构**：

```python
'require': ['customer'],  # 依赖单个模块
'require': ['customer', 'supplier'],  # 依赖多个模块
'require': [],  # 无依赖
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `require` | list[string] | 否 | 依赖模块的 ID 列表 |

**验证逻辑**：

| 时机 | 说明 |
|------|------|
| 安装时 | 检查所有依赖模块是否已安装并启用 |
| 启用时 | 检查所有依赖模块是否已安装并启用 |

**验证流程**：

```
安装/启用模块 A
        │
        ▼
    读取 A.require
        │
        ▼
    require 为空或不存在？
    ├── 是 → 验证通过，允许操作
    └── 否 → 进入依赖检查
        │
        ▼
    遍历 require 列表
        │
        ├── 依赖模块 B 是否已安装+启用？
        │   ├── 否 → 返回错误，阻止操作
        │   └── 是 → 继续检查 B 的依赖（递归）
        │
        ▼
    所有依赖验证通过
        │
        ▼
    允许安装/启用
```

**错误提示示例**：

```
无法安装模块「模块A」

原因：需要「模块B」已安装并启用（当前状态：未安装）
```

**循环依赖检测**：

系统会检测并阻止循环依赖，例如 A→B→A：

```
无法安装模块「模块A」

原因：发现循环依赖

依赖链：模块A → 模块B → 模块A

请检查模块的 require 配置，移除循环依赖。
```

**向后兼容**：

| 场景 | 处理方式 |
|------|----------|
| `require` 属性不存在 | 视为无依赖，检查通过 |
| `require` 为空列表 `[]` | 视为无依赖，检查通过 |
| 依赖模块不存在于系统中 | 跳过检查（允许第三方模块依赖外部模块） |

## 五、模块管理

### 5.1 模块生命周期

```
未注册 → 注册 → 安装 → 启用
              ↓
            禁用
```

**安装流程详细步骤**：

```
注册 → 安装：
  1. 检查模块目录是否存在
  2. 检查 migrations 目录是否存在
  3. 验证表是否存在（如 is_installed=True 但表不存在，取消标记）
  4. 执行 makemigrations（生成迁移文件）
  5. 执行 migrate（应用迁移）
  6. 验证表是否创建成功
  7. 如验证失败：抛出 RuntimeError
  8. 如验证成功：标记 is_installed=True
  9. 同步 NodeType（仅 node 类型）
  10. 创建模块词汇表
```

### 5.2 模块状态

| 状态 | 说明 | 触发操作 |
|------|------|----------|
| **未注册** | 目录存在但未注册 | 扫描模块 |
| **已注册** | 已注册到数据库 | 注册模块 |
| **已安装** | 数据库迁移已完成 | 安装模块 |
| **已启用** | 可正常使用 | 启用模块 |
| **已禁用** | 已安装但未启用 | 禁用模块 |

### 5.3 类型过滤逻辑

| 场景 | `type='node'` | `type='system'` | `type='tool'` |
|------|---------------|-----------------|-----------------|
| 扫描注册 | ✅ | ✅ | ✅ |
| 安装模块 | 创建 NodeType | ❌ | 创建 ToolType |
| 启用模块 | 启用节点类型 | ❌ | 启用工具类型 |
| 事务首页 | ✅ | ❌ | ❌ |
| 协作工具首页 | ❌ | ❌ | ✅ |

### 5.4 管理功能

| 功能 | 说明 |
|------|------|
| 扫描模块 | 使用 AST 直接解析 module.py，无需重启服务 |
| 安装模块 | 验证依赖 → 执行迁移 → 创建 NodeType/ToolType → 创建词汇表 |
| 启用模块 | 验证依赖 → 启用节点类型（node）或工具类型（tool） |
| 禁用模块 | 禁用节点类型（node）或工具类型（tool） |

### 5.5 依赖检查

模块安装和启用时，系统会自动检查 `require` 字段中声明的依赖是否满足。

**检查时机**：

| 操作 | 是否检查依赖 |
|------|-------------|
| 安装模块 | ✅ 检查 |
| 启用模块 | ✅ 检查 |
| 禁用模块 | ❌ 不检查 |
| 卸载模块 | ❌ 不检查 |

**检查要求**：

- 依赖模块必须已安装（`is_installed=True`）
- 依赖模块必须已启用（`is_active=True`）
- 支持递归检查（A→B→C 的依赖链）
- 支持循环依赖检测

**相关服务方法**：

| 方法 | 说明 |
|------|------|
| `ModuleService.check_dependencies()` | 递归检查模块依赖 |
| `ModuleService.verify_dependencies()` | 验证依赖是否满足 |
| `ModuleService.get_dependency_chain()` | 获取完整依赖链 |

### 5.6 错误处理规范

模块安装过程中的错误处理必须遵循以下规范：

| 场景 | 处理方式 |
|------|----------|
| 迁移命令失败 | 收集错误信息，抛出 `RuntimeError` |
| 表不存在但标记已安装 | 取消 `is_installed` 标记，重新执行迁移 |
| 迁移后表未创建 | 抛出异常，提示模块配置问题 |
| 模块目录不存在 | 返回 `False`，不执行后续操作 |

**核心原则**：迁移失败必须抛出异常，不能静默跳过。

**错误处理示例**：

```python
# 安装模块时的错误处理
migration_errors = []

try:
    call_command('makemigrations', module_id, verbosity=1, interactive=False)
except Exception as e:
    migration_errors.append(f'makemigrations 失败: {e}')

try:
    call_command('migrate', module_id, verbosity=1, interactive=False, run_syncdb=True)
except Exception as e:
    migration_errors.append(f'migrate 失败: {e}')

# 验证迁移是否成功
if not check_tables_exist(module_id):
    migration_errors.append(f'迁移后表仍未创建，模块 {module_id} 可能配置不正确')

# 如果有迁移错误，抛出异常
if migration_errors:
    raise RuntimeError(f'模块 {module_id} 安装失败: {"; ".join(migration_errors)}')
```

### 6.3.1 动态加载机制

> **重要**：为确保系统在任何模块缺失时都能正常启动，modules/urls.py 必须是**动态安全加载**的。

使用动态加载机制后：
- 模块文件夹被删除时，系统仍能正常启动
- 模块被禁用时，不影响系统运行
- 无需手动更新 URL 路由表，模块自动从数据库加载

**实现代码**：

```python
# modules/urls.py
from django.urls import path, include
from importlib import import_module

def try_include_module(module_slug):
    """尝试动态导入模块 URL，失败则返回空列表"""
    try:
        import_module(f'modules.{module_slug}.urls')
        return [path(f'{module_slug}/', include(f'modules.{module_slug}.urls'))]
    except (ImportError, ModuleNotFoundError, AttributeError):
        return []

def get_installed_module_slugs():
    """动态获取所有已安装模块的 slug"""
    try:
        from core.node.models import Module
        modules = Module.objects.filter(is_installed=True, is_active=True)
        return [m.module_id for m in modules]
    except Exception:
        return []

# 动态加载所有已安装模块
_installed_slugs = get_installed_module_slugs()
_dynamic_routes = []
for _mod in _installed_slugs:
    _dynamic_routes.extend(try_include_module(_mod))

# 基础 URL 路由（不变）
_base_urlpatterns = [
    path('', node_views.nodes_index, name='index'),
    # 字段类型、导入导出等固定路由...
]

# 合并所有路由
urlpatterns = _base_urlpatterns + _dynamic_routes
```

**注意事项**：
1. 模块从数据库动态加载，无需手动维护列表
2. 通用节点路由（`<slug:node_type_slug>/`）必须放在动态路由**之后**，作为 fallback
3. 动态加载失败时使用 `except (ImportError, ModuleNotFoundError, AttributeError)` 捕获所有异常

### 8.3 安装时的迁移验证

模块安装时，系统必须验证迁移是否成功，确保数据库状态与模块注册状态一致。

**验证步骤**：

| 步骤 | 说明 |
|------|------|
| 1 | 检查 `django_migrations` 表中的迁移记录 |
| 2 | 验证模块所有模型对应的表是否存在 |
| 3 | 如验证失败，执行错误处理 |

**表存在性检查示例**：

```python
from django.apps import apps
from django.db import connection

def check_tables_exist(module_id: str) -> bool:
    """检查模块的表是否已存在"""
    try:
        models = apps.get_app_config(module_id).get_models()
        for model in models:
            table_name = model._meta.db_table
            with connection.cursor() as cursor:
                cursor.execute(
                    "SELECT name FROM sqlite_master WHERE type='table' AND name=?",
                    [table_name]
                )
                if not cursor.fetchone():
                    return False
        return True
    except Exception:
        return False
```

**状态一致性保证**：

如果 `Module.is_installed=True` 但表不存在，说明之前的迁移可能失败过。系统应：
1. 取消 `is_installed` 标记
2. 重新执行迁移
3. 验证迁移成功后再标记

## 十一、功能卡片系统

### 11.1 概述

功能卡片是首页仪表盘上的快捷入口区域，采用 3×2 网格布局（6个槽位），用于快速跳转到各模块页面。

**与导航卡片的区别**：

| 特性 | 功能卡片 | 导航卡片 |
|------|----------|----------|
| 位置 | 首页顶部 | 首页中部 |

| 来源 | 模块配置 `dashboard_cards` | 用户自定义 |

```html
        <div class="fs-5 fw-bold mb-2">供应商</div>
        <div class="fs-1 fw-bold">{{ total|default(0) }}</div>
        <div class="small opacity-75">本周新增 {{ recent|default(0) }} 条</div>
    </div>
</div>
```

**卡片模板规范**：

1. **根元素**：使用 `<div>`，不是 `<a>`
2. **必需 class**：`module-card`、`h-100`（高度填满容器）
3. **必需属性**：
   - `draggable="true"` - 支持拖拽
   - `data-module="{module_id}"` - 标识模块
4. **背景色**：通过内联 `style` 定义，使用渐变色
5. **内边距**：使用 `p-3` 或 `p-4`
6. **内容居中**：`d-flex flex-column justify-content-center align-items-center text-center`

### 11.8 功能卡片链接规范

功能卡片需要正确链接到模块列表页，必须遵循以下规范：

**链接路径格式**：

```javascript
card.href = '/modules/' + moduleId + '/';
```

**模块名称映射**：

在 `dashboard.html` 的 `createCustomerCard` 函数中，通过 `moduleNames` 对象定义模块显示名称：

```javascript
var moduleNames = {
    'customer': '海外客户',
    'customer_cn': '国内客户',
    'new_module': '新模块名称'  // 添加新模块
};
```

**完整示例**：

```javascript
function createCustomerCard(moduleId, position, stats) {
    var moduleNames = {
        'customer': '海外客户',
        'customer_cn': '国内客户',
        'new_module': '新模块'
    };
    var moduleName = moduleNames[moduleId] || moduleId;
    var total = stats ? stats.total : 0;
    var recent = stats ? stats.recent : 0;
    
    var card = document.createElement('a');
    card.href = '/modules/' + moduleId + '/';  // 必须使用此格式
    card.className = 'module-card';
    // ... 其他样式设置
    
    card.innerHTML = 
        '<div class="card-title">' + moduleName + '</div>' +
        '<div class="card-count">' + total + '</div>' +
        '<div class="card-subtitle">本周新增 ' + recent + ' 条</div>';
    
    return card;
}
```

**注意事项**：

1. 链接路径必须以 `/modules/` 开头
2. 模块名称映射必须在 `moduleNames` 对象中定义
3. 新增模块时需同步更新 `moduleNames` 对象

### 11.9 模板预渲染机制

系统会在 API 返回时自动预渲染模块的 `dashboard_card.html` 模板，生成的 HTML 存储在 `module_contents` 字段中。

**预渲染流程**：

```
API 请求 /api/user/dashboard/cards/
         │
         ▼
    遍历已激活模块
         │
         ▼
    检查 module_info['dashboard_cards']
         │
         ├── 无配置 → 跳过
         │
         └── 有配置 → 加载模板
                     │
                     ▼
                 Jinja2 渲染（传入动态变量如 wa_connected）
                     │
                     ▼
                 存入 module_contents
```

**模板路径规则**：

- 路径相对于模块的 `templates/` 目录
- 例如：`whatsapp/dashboard_card.html` 对应文件 `modules/whatsapp/templates/whatsapp/dashboard_card.html`
- Django 会自动搜索 `modules/*/templates/` 目录

**模板变量**：

| 变量 | 说明 |
|------|------|
| `module_id` | 模块 ID |
| `stats` | 统计数据对象（包含 `total`、`recent`） |
| `wa_connected` | WhatsApp 连接状态（仅 whatsapp 模块） |

**模板示例**：

```html
<!-- whatsapp 模块示例 -->
<div class="h-100 module-card" draggable="true" data-module="whatsapp"
     style="background: linear-gradient(135deg, #25D366 0%, #128C7E 100%); color: white; border: none;">
    <div class="d-flex flex-column justify-content-center align-items-center text-center p-3 h-100">
        <div class="fs-5 fw-bold mb-2">📱 WhatsApp 发送</div>
        <div class="text-center">
            {% if wa_connected %}
            <span class="badge bg-light text-success">● 已连接</span>
            {% else %}
            <span class="badge bg-light text-danger">○ 未连接</span>
            {% endif %}
        </div>
    </div>
</div>
```

**前端使用**：

前端直接使用 `module_contents` 中的预渲染 HTML，无需再执行客户端渲染。

## 十二、协作工具页面（Tool Dashboard）

### 12.1 概述

协作工具页面是专用于 `tool` 类型模块的展示区域，采用与事务处理页面类似的架构，提供卡片式首页和左侧导航。

**核心特性**：
- `tool` 类型模块自动显示在协作工具首页
- 支持左侧动态导航
- 采用模块化动态加载机制

### 12.2 架构

协作工具页面完整架构：

```
顶部菜单栏 → 协作工具入口（/tools/）
                ↓
         tools_index 视图
                ↓
         tools_dashboard.html（工具总览页面）
                ↓
         frame_tools.html（框架模板）
                ↓
         tools_page 视图（动态加载工具模块）
```

| 层级 | 文件 | 说明 |
|------|------|------|
| 入口 | `core/templates/includes/nav.html` | 链接到 `core:tools_index` |
| 路由 | `core/urls.py` | `/tools/` 和 `/tools/<slug>/` |
| 视图 | `core/views/tools.py` | `tools_index`, `tools_page` |
| 框架 | `core/templates/frames/frame_tools.html` | 侧边栏 + 主内容区 |
| 内容 | `core/templates/tools/tools_dashboard.html` | 工具卡片列表 |

### 12.3 工具模块配置

`tool` 类型模块在 `module.py` 中配置：

#### 12.7.1 views.py 视图函数规范

工具模块的视图函数必须遵循以下规范，否则侧边栏的工具链接将无法正常显示：

```python
# modules/calc/views.py
from django.shortcuts import render
from django.contrib.auth.decorators import login_required
from core.node.models import Module, ToolType


@login_required
def tool_view(request):
    """计算器工具页面"""
    # ⚠️ 必须传递 tools 变量，否则 frame_tools.html 侧边栏的工具链接会消失
    tool_type_ids = Module.objects.filter(
        module_type='tool',
        is_active=True
    ).values_list('module_id', flat=True)
    tools = ToolType.objects.filter(slug__in=tool_type_ids, is_active=True)
    
    return render(request, 'calc/calc.html', {
        'tools': tools,
    })
```

**关键点**：
- `tools_page` 视图会自动传递 `tools` 变量给模板
- 但模块自己的视图（如 `tool_view`）需要手动查询并传递
- 如果不传递 `tools`，`frame_tools.html` 的侧边栏只会显示"工具总览"，不会显示其他已启用的工具链接

#### 12.7.2 模板继承规范

工具模块的模板必须继承 `frames/frame_tools.html`，并设置正确的 `active_section`：

```html
{% extends "frames/frame_tools.html" %}

{% set show_header = false %}
{% set active_section = 'calc' %}

{% block admin_title_content %}
  <i class="bi bi-calculator me-2"></i>计算器
{% endblock %}

{% block admin_content %}
  <!-- 页面内容 -->
{% endblock %}
```

- `active_section` 必须设置为模块的 slug（如 `calc`），用于侧边栏高亮当前页面

#### 12.7.3 无 models.py 的模块安装

工具类型模块可能没有数据库模型（如计算器模块），`ModuleService` 已处理这种情况：

1. 检查模块是否有 `models.py` 文件
2. 如果没有，跳过迁移步骤，直接标记为已安装
3. 仍然会同步创建 `ToolType`

```python
# module_service.py 中的处理逻辑
models_path = os.path.join(module_path, 'models.py')
has_models = os.path.exists(models_path)

if not has_models:
    # 跳过迁移，直接标记安装成功
    pass
elif has_migrations:
    # 执行迁移
    ...
```

这意味着你可以创建纯前端的工具模块，无需创建数据库表。

| 场景 | `type='node'` | `type='system'` | `type='tool'` |
|------|---------------|-----------------|-----------------|
| 扫描注册 | ✅ | ✅ | ✅ |
| 安装模块 | 创建 NodeType | ❌ | 创建 ToolType |
| 启用模块 | 启用节点类型 | ❌ | 启用工具类型 |
| 事务首页 | ✅ | ❌ | ❌ |
| 协作工具首页 | ❌ | ❌ | ✅ |
| 事务页面侧边栏 | ✅ | ❌ | ❌ |
| 协作工具侧边栏 | ❌ | ❌ | ✅ |
