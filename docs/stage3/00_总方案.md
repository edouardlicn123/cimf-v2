# Django 迁移总方案

## 项目概述

| 项目 | 说明 |
|------|------|
| 项目名称 | CIMF - Corporate Internal Management Framework |
| 目标 | 使用 Django 框架完全重新实现 Flask 项目功能 |
| 原项目 | 已备份到 `cimf-flask/` |
| 数据库 | SQLite |

---

## 技术栈

| 类别 | 技术 |
|------|------|
| 后端 | Python 3.12 / Django 6.0 |
| API | Django REST Framework |
| 数据库 | SQLite |
| 模板 | Jinja2（兼容现有模板） |
| 认证 | Django Auth |

## 关键技术决策

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 用户模型 | AbstractUser 扩展 | 保留现有字段 |
| 模板引擎 | Jinja2 | 复用现有模板 |
| 管理后台 | Django Admin + 自定义 | 快速开发 |
| 定时任务 | Django Management Command | 与 CronService 兼容 |
| 数据库 | SQLite | 无需迁移 |

---

## 数据迁移

由于使用 SQLite，新 Django 项目可以直接访问现有数据库文件 `instance/site.db`。

如需完全迁移：
1. 导出 Flask 数据为 JSON/SQL
2. 运行 Django migrations
3. 导入数据

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 模板兼容性 | 中 | Jinja2 后端配置 |
| 认证差异 | 高 | Django Auth 直接使用 |
| 字段类型差异 | 中 | 自定义字段映射 |
