# 20_客户模块目录重构计划（修订版）

## 概述

重新梳理项目架构，将 Node 支撑系统移入 core 目录，nodes 目录只保留具体业务模块。

## 架构设计理念

| 目录 | 职责 |
|------|------|
| **core/** | 系统级基础功能框架 |
| **core/node/** | Node 节点系统框架（通用底层，不含业务） |
| **nodes/** | 具体业务模块（客户、会员、项目等） |

## 背景

当前项目结构存在以下问题：

1. **Node 支撑系统与业务模块混在一起**：`nodes/models.py` 包含 `NodeType`、`Node` 这样的通用框架模型，也包含 `CustomerFields` 这样的业务模型
2. **服务层职责不清晰**：`nodes/services/` 目录既有通用服务（`NodeService`、`NodeTypeService`），也有业务服务（`CustomerService`）
3. **扩展性受限**：如果要在 nodes 下添加新的业务模块（如会员、项目），不够清晰

## 实施步骤

### 步骤 1: 创建 core/node/ 目录结构

```
core/node/
├── __init__.py
├── models.py
├── services.py
├── views.py
├── urls.py
├── admin.py
└── forms.py
```

### 步骤 2: 移动 Node 支撑系统文件

1. 创建 `core/node/models.py`，将 `NodeType` 和 `Node` 类从 `nodes/models.py` 移入
2. 合并服务文件到 `core/node/services.py`（合并 `NodeTypeService`、`NodeService` 等）
3. 移动 `nodes/forms.py` 中的 `NodeTypeForm` 到 `core/node/forms.py`
4. 移动 `nodes/admin.py` 中的 `NodeTypeAdmin`、`NodeAdmin` 到 `core/node/admin.py`
5. 移动 `nodes/views.py` 中的节点类型管理视图到 `core/node/views.py`
6. 移动 `nodes/urls.py` 到 `core/node/urls.py`

### 步骤 3: 更新 nodes/models.py

更新导入路径，从 core.node.models 导入 NodeType 和 Node：

```python
# nodes/models.py
from core.node.models import NodeType, Node

# 保留业务模型
class CustomerFields(models.Model):
    node = models.OneToOneField(
        Node,  # 使用导入的 Node
        on_delete=models.CASCADE,
        ...
    )
```

### 步骤 4: 更新 nodes/views.py

- 删除节点类型管理视图（已移至 core/node/views.py）
- 更新导入路径，从 core.node 导入服务层

```python
# nodes/views.py
# 节点类型管理视图已移除
# 客户相关视图保留

from core.node.services import NodeTypeService, NodeService
from nodes.customer.services import CustomerService
from nodes.customer_cn.services import CustomerCnService
```

### 步骤 5: 更新 nodes/urls.py

更新导入路径：

```python
# nodes/urls.py
from core.node.views import ...  # 节点管理视图
```

## 预期效果

1. **架构清晰**：核心框架（Node）与业务模块分离，各自独立维护
2. **职责明确**：`core/node/` 负责通用节点管理，`nodes/` 负责具体业务实现
3. **扩展性提升**：在 nodes 下添加新业务模块更加清晰
4. **解耦优化**：业务模块之间互不依赖，可独立演进

## 风险与注意事项

1. **数据库迁移**：NodeType 和 Node 模型移动后需要创建迁移，可能涉及数据迁移
2. **循环导入**：注意 core.node 和 nodes 之间的依赖关系，避免循环导入
3. **测试覆盖**：修改后需运行完整测试
4. **URL 路由调整**：需要确保旧 URL 路径仍然有效或提供重定向
5. **Admin 配置**：Django Admin 需要重新注册模型
6. **命名冲突**：注意避免与 core 现有文件重名
